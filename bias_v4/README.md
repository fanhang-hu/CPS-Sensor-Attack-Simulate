This experiment achieves the bias attack by performing **Memory Tampering** at the ```controller```,
- The ```sensor``` normally sends data to the ```controller```.
- The attack achieves bias tampering by modifying the measurement values in the controller's memory.
- Under attack, the attack process will naturally generate abnormal system calls ```ptrace, process_vm_readv, process_vm_writev```.

First, ```attacker_bias.c```,
- By using ```ptrace``` and the cross-process memory read-write system call ```process_vm_writev```, a double variable (g_latest_measurement) in the memory of the ```controller``` is tampered with during its operation. Thus, a fixed bias is introduced in the calculation of the controller. **```ptrace``` can temporarily obtain operation permissions and avoid data inconsistency.**

The prerequisite for a successful attack,
- ```The same user ID:``` The attacker must be running under the same user.
- ```ptrace_scope configuration:``` If the system enables ```Yama`` and ```ptrace_scope``` is set to 1, it needs to be pre-set to 0. The controller calls ```prctl(PR_SET_PTRACER, PR_SET_PTRACER_ANY)```.
- The attacker obtains the addresses of ```PID``` and ```g_latest_measurement``` by reading the ```meta-file (/tmp/cps_controller_meta.txt)``` generated by the controller.
- A delay ```(attack_window_ms)``` between the controller receiving data and using it. The attacker must complete additional, read and write, and separate operations within this window. Controller latency is typically tens of milliseconds, which is sufficient for attackers to complete operations (both ptrace add-ons and read-write operations are at the microsecond level).

To sum up,
- Each time the ```controller``` receives new data from the sensor, it immediately stores the value in ```g_latest_measurement``` and then **sleeps attack_window_ms**.
- The attacker tampered with the variable during sleep, changing its value to the original value plus bias.
- After waking up, the controller reads g_latest_measurement and uses the tampered value to calculate the control output, resulting in control bias.

## Weakness
The unreasonable or simplified aspects existing in the experiment:
- Attackers can easily obtain the address of the target variable. The ```controller``` actively writes the address of ```g_latest_measurement``` to ```/tmp/cps_controller_meta.txt```, and **the attacker can directly read this file to obtain the critical information**. While in real systems, variable addresses are usually not exposed in plaintext form; Even if the attacker acquires the PID, precisely locating the memory address of the target variable still requires more complex reverse engineering or information leakage vulnerabilities. In addition, ```ASLR``` (Address Space Layout Randomization) makes the address of each run different, making it difficult for attackers to predict in advance.

- The controller actively relaxes the ptrace permission. The ```controller``` code calls: ```prctl(PR_SET_PTRACER, PR_SET_PTRACER_ANY)```, this allows any process to track the controller, which is equivalent to opening the door for attackers. This configuration is usually not available in the production environment. The default Yama ptrace_scope value is 1 (only allowing child process tracking) or higher, and attackers cannot directly attach ptrace_to non-child processes. ***Unless the attacker already has root privileges or the system has been deliberately weakened***, this attack will not succeed.

- The attacker precisely controls the attack time window. The ```attacker``` assumes that the delay window of the controller is fixed (such as 80 ms), and that their own attack interval (such as 100 ms) matches it, ```ensuring that every tampering occurs within the window```. **The processing delay of real systems may change dynamically**, making it difficult for attackers to synchronize precisely. In addition, it takes time for attackers to attach/separate ```ptrace``` itself (in the microsecond range, but there is still uncertainty), which may cause them to miss the window or have the target process suspended for too long, and be detected by the system or administrator.

- The attacker and the target process are running under the same user. By default, **the attacker and the controller in the experiment are started by the same user**, so ```process_vm_writev``` does not require additional permissions. While in industrial control systems, controllers often operate as ```dedicated low-privilege users```. If attackers do not obtain the privileges of this user, they cannot use ```ptrace``` or ```process_vm_writev```. **If the attacker has obtained the user's permission and the system has been severely invaded, the attack methods are diverse and memory tampering may not necessarily be chosen**.

- The attacker knows the internal parameters of the system. The ```attacker``` needs to know the controller's PID, variable address, attack window length, etc. In the experiment, this information is passed through **meta-files and environment variables**. While in real attacks, attackers need to first obtain this information through vulnerabilities, and the difficulty of information collection is greatly underestimated.
