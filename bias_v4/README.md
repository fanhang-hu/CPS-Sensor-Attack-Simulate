This experiment achieves the bias attack by performing **Memory Tampering** at the ```controller```,
- The ```sensor``` normally sends data to the ```controller```.
- The attack achieves bias tampering by modifying the measurement values in the controller's memory.
- Under attack, the attack process will naturally generate abnormal system calls ```ptrace, process_vm_readv, process_vm_writev```.

First, ```attacker_bias.c```,
- By using ```ptrace``` and the cross-process memory read-write system call ```process_vm_writev```, a double variable (g_latest_measurement) in the memory of the ```controller``` is tampered with during its operation. Thus, a fixed bias is introduced in the calculation of the controller. **```ptrace``` can temporarily obtain operation permissions and avoid data inconsistency.**

The prerequisite for a successful attack,
- ```The same user ID:``` The attacker must be running under the same user.
- ```ptrace_scope configuration:``` If the system enables ```Yama`` and ```ptrace_scope``` is set to 1, it needs to be pre-set to 0. The controller calls ```prctl(PR_SET_PTRACER, PR_SET_PTRACER_ANY)```.
- The attacker obtains the addresses of ```PID``` and ```g_latest_measurement``` by reading the ```meta-file (/tmp/cps_controller_meta.txt)``` generated by the controller.
- A delay ```(attack_window_ms)``` between the controller receiving data and using it. The attacker must complete additional, read and write, and separate operations within this window. Controller latency is typically tens of milliseconds, which is sufficient for attackers to complete operations (both ptrace add-ons and read-write operations are at the microsecond level).

To sum up,
- Each time the ```controller``` receives new data from the sensor, it immediately stores the value in ```g_latest_measurement``` and then **sleeps attack_window_ms**.
- The attacker tampered with the variable during sleep, changing its value to the original value plus bias.
- After waking up, the controller reads g_latest_measurement and uses the tampered value to calculate the control output, resulting in control bias.

## Experiment Step
It needs to be compiled first,
```
cd /home/hfh/cps_bias_v4
sudo make clean && sudo make
```

First of all, run **NoDrop**,
```
cd /home/hfh/NoDrop/build
sudo insmod ./nodrop.ko
```

In another terminal,
```
./scripts/run_experiment.sh --mode baseline --duration-sec 30
./scripts/run_experiment.sh --mode bias --duration-sec 30
```

Finally, stop NoDrop,
```
sudo rmmod nodrop
```

In /tmp/nodrop, you can see the monitor buf.
